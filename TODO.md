===============
Сейчас логика каталога явно перегружает `SettingsService` ([SettingsService.java](/Users/nick/IdeaProjects/human-guided-development/backend/src/main/java/ru/hgd/sdlc/settings/application/SettingsService.java)): там и runtime settings, и git/mirror, и repair/upsert, и валидация контента.

Что даст `CatalogService`:
1. Чёткие границы ответственности (`settings` vs `catalog ingest/sync/repair`).
2. Проще тестировать и эволюционировать каталог отдельно.
3. Меньше риск регрессий в runtime settings при изменениях каталога.
4. Нормальная точка для единой политики валидации flow/rule/skill.

Как я бы декомпозировал:
1. `CatalogService`: orchestrator sync/repair, обход mirror, dispatch по entity type.
2. `CatalogValidationService`: schema/frontmatter/yaml валидация.
3. `CatalogUpsertService`: upsert rule/skill/flow + checksum/идемпотентность.
4. `CatalogGitService`: clone/fetch/reset, работа с mirror/workdir.

Риски:
1. Не разнести транзакционные границы (repair lock + DB tx + git операции).
2. Появление дублирующей валидации с `FlowYamlParser`/markdown parser.
3. Сломать обратную совместимость API `settings/catalog/repair`.

Итог: **поддерживаю**, это правильный рефакторинг. Делать лучше поэтапно, начиная с выделения `CatalogService` как фасада без изменения внешних API, затем выносить внутренние куски.
