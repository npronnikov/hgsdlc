# Product Requirements Document - hgsdlc

**Author:** HGS Team
**Date:** 2026-03-27

## Executive Summary

hgsdlc (Human Guided SDLC) — внутренняя B2B SaaS платформа управляемой автономии разработки ПО для fintech-команд. Решает фундаментальную проблему существующих AI-инструментов: недетерминированность выполнения процессов — пропуск шагов, некорректная интерпретация инструкций, вариативность результатов от запуска к запуску. Обеспечивает повторяемые, контролируемые и аудируемые AI-воркфлоу по полному SDLC-циклу: от требований до кода и тестов.

**Целевые пользователи:** продуктовые команды fintech-компаний — архитекторы, аналитики, разработчики, тимлиды — которым требуется трассируемость артефактов и воспроизводимость процесса для соответствия внутренним compliance-требованиям.

### Что делает это особенным

**Гарантированное соответствие процессу.** Детерминированность достигается через версионную фиксацию всех компонентов выполнения: воркфлоу, скиллы, rules и модель LLM фиксируются до запуска. Идентичный набор версий = структурно эквивалентный выход каждый раз. Аналогия: `package-lock.json` для AI-процессов разработки.

**Прогрессивная автономия.** Система накапливает доверие: по мере роста числа успешных запусков объём ручных подтверждений снижается — от пошагового контроля до zero-approval автономии. Накопленная история выполнения — несокопируемый актив команды.

**Audit-first архитектура.** Каждый выходной артефакт (спецификация, код, тест) связан с версией воркфлоу, скиллов и rules, которые его породили. Полная трассируемость по умолчанию.

## Success Criteria

### Пользовательский успех

- Первый запуск воркфлоу генерирует спецификацию, принятую командой без доработок
- Сгенерированный код соответствует спецификации и проходит функциональную проверку с первой попытки
- Команда воспринимает результат воркфлоу как надёжный и предсказуемый

### Бизнес-успех

- **3 месяца:** снижение lead time разработки на 10% относительно baseline до внедрения
- **12 месяцев:** снижение lead time разработки на 50% относительно baseline до внедрения

### Технический успех

- 95% запусков воркфлоу завершаются без ручных исправлений выходных артефактов
- 100% действий в воркфлоу покрыты audit log
- Идентичные входные параметры и версии компонентов дают структурно эквивалентные выходные артефакты

### Измеримые результаты

| Метрика | Цель | Горизонт |
|---|---|---|
| Артефакты без ручных исправлений | ≥ 95% запусков | Постоянно |
| Снижение lead time | 10% | 3 месяца |
| Снижение lead time | 50% | 12 месяцев |
| Audit log покрытие | 100% действий | MVP |

## Product Scope

### MVP — Phase 1

| Возможность | Обоснование |
|---|---|
| Движок выполнения воркфлоу с версионной фиксацией | Ядро продукта — без него нет ценности |
| Каталог воркфлоу (шаблоны) | Без каталога команды не могут начать работу |
| Пошаговое выполнение с подтверждением | Основной UX первых пользователей |
| Audit log всех действий | Требование для fintech-команд |
| Git self-hosted интеграция | GitLab CE/EE, Bitbucket Server |
| Dashboard метрик (базовый) | Видимость прогресса для Team Lead |
| RBAC (6 ролей) + Workspace-изоляция | Безопасность и разделение данных |
| Базовая аутентификация (username/password) | LDAP/SAML откладывается в Growth |

**MVP Strategy:** Platform MVP — стабильность движка приоритетнее полноты фич. Первые пользователи — внутренние команды компании (контролируемая среда, без внешних обязательств). Разработка с нуля — начать с движка на 2-3 шага, валидировать детерминированность, затем масштабировать.

### Growth — Phase 2

- **LDAP/AD, SAML SSO** — корпоративная аутентификация
- **Прогрессивная автономия** — auto-approve на основе истории успешных запусков
- **Git публичные провайдеры** — GitHub, GitLab.com
- **CI/CD интеграция** — триггеры воркфлоу из пайплайна
- **Сравнение артефактов** — diff между запусками одного воркфлоу

### Vision — Phase 3

- **Полная автономия** — zero human approvals на основе накопленного доверия
- **Адаптивные воркфлоу** — самонастройка параметров по истории команды
- **Экосистема каталога** — marketplace воркфлоу от сообщества fintech-команд

### Risk Mitigation

| Риск | Стратегия |
|---|---|
| Движок с нуля — высокая неопределённость | Минимальный движок (2-3 шага) → валидация → масштабирование |
| Нет внешней валидации спроса | Пилот с 1-2 внутренними командами, измерять lead time до/после |
| Ресурсов меньше плана | Приоритет: движок + каталог + audit log; dashboard и LDAP откладываются |

## User Journeys

### Journey 1: Аналитик запускает воркфлоу — успешный путь

**Персонаж:** Алина, старший бизнес-аналитик в fintech-компании.

**Открывающая сцена:** Бизнес поставил задачу: новый модуль KYC-верификации. Раньше Алина проводила 2-3 встречи, писала спецификацию вручную, затем переписывала после ревью архитектора.

**Развитие:** Алина выбирает воркфлоу "Создание функциональной спецификации" из каталога, вводит параметры задачи. Система проходит шаги: анализ контекста → формирование требований → генерация спецификации. На каждом шаге Алина видит промежуточный артефакт и подтверждает или корректирует.

**Кульминация:** Через 20 минут спецификация готова. Архитектор: "Это можно брать в работу без доработок."

**Развязка:** Спецификация зафиксирована в Git с полным audit log. Алина сохраняет 4 часа итераций.

**Раскрытые требования:** FR1, FR5–FR8, FR10–FR13, FR15, FR17, FR32, FR39

### Journey 2: Разработчик получает и реализует артефакты

**Персонаж:** Дмитрий, senior backend-разработчик. Боль: "Написано одно, имелось в виду другое."

**Открывающая сцена:** Алина завершила воркфлоу. Дмитрий получает in-app уведомление: спецификация готова.

**Развитие:** Дмитрий запускает воркфлоу "Генерация кода по спецификации". Система генерирует код, тесты, документацию API — каждый артефакт трассирован к требованию спецификации.

**Кульминация:** Тесты проходят. Diff между кодом и спецификацией — полное соответствие. Не нужно "додумывать" детали реализации.

**Развязка:** Код в Git. В dashboard виден путь: задача → спецификация → код → тесты. Lead time — 3 часа вместо 2 дней.

**Раскрытые требования:** FR1, FR3, FR5–FR13, FR24–FR25, FR28–FR29, FR38

### Journey 3: Team Lead мониторит прогресс и метрики

**Персонаж:** Максим, Tech Lead команды из 8 человек.

**Открывающая сцена:** Пятница. Максим хочет понять состояние процессов без погружения в детали.

**Развитие:** Открывает dashboard. Видит: 12 воркфлоу за неделю, 11 без ручных исправлений (92%), 1 с корректировкой на шаге архитектурного ревью. Lead time по спецификациям снизился на 35%.

**Кульминация:** Максим видит паттерн — воркфлоу архитектурного ревью регулярно требует вмешательства. Проблема в шаблоне воркфлоу.

**Развязка:** Dashboard стал основой ретроспективы. Прогресс в цифрах, а не на словах.

**Раскрытые требования:** FR24–FR27

**Развитие:** Сергей создаёт workspace, назначает роли, открывает доступ к воркфлоу из каталога. Новая команда видит только свои данные.

**Кульминация:** Новая команда запускает первый воркфлоу на следующий день после онбординга — без обращений в поддержку.

### Риски домена

| Риск | Митигация |

## Innovation & Novel Patterns

### Инновационные паттерны

**1. Immutable Workflow Execution (версионная фиксация)**

Детерминированность через иммутабельность: воркфлоу, скиллы, rules и модель LLM фиксируются до запуска. Идентичный набор версий гарантирует структурно эквивалентный выход независимо от даты и модели.

**2. Прогрессивная автономия как трастовый механизм**

Система динамически снижает необходимость в подтверждениях по мере накопления истории успешных запусков. Накопленная история выполнения — конкурентный актив команды, не воспроизводимый конкурентами.

**3. Audit-first архитектура AI-разработки**

Трассируемость как архитектурный принцип: каждый артефакт содержит ссылку на полный контекст генерации. По артефакту можно точно восстановить условия его создания.

### Рыночный контекст

Аналогов не обнаружено. Существующие подходы (промпт-инжиниринг, AI в CI/CD) не обеспечивают версионную фиксацию всего контекста выполнения как системный механизм воспроизводимости.

### Валидация и риски инновации

- **MVP-метрика:** ≥ 95% запусков без ручных исправлений артефактов
- **Baseline:** первые 3 команды фиксируют baseline для сравнения запусков

| Риск | Митигация |
|---|---|
| LLM-провайдер меняет модель | Версия модели — часть конфигурации воркфлоу |
| Жёсткая фиксация тормозит обновления | Механизм миграции с changelog и diff-тестированием |
| "Детерминированность" = "плохой вывод" | Метрика качества артефактов отдельно от воспроизводимости |

## SaaS B2B — Специфические требования

### Модель мультитенантности

**Workspace-модель:** каждая команда — изолированный workspace. Команды не имеют доступа к данным друг друга. Platform Admin управляет всеми workspace'ами.

- Один workspace = одна команда
- Каталог воркфлоу: два уровня — организационный (Platform Admin) и workspace (Team Lead)
- Изоляция на уровне хранилища (tenant_id обязателен для всех записей)

### Матрица ролей и прав (RBAC)

| Действие | Platform Admin | Team Lead | Architect | Analyst | Developer | Auditor |
|---|:---:|:---:|:---:|:---:|:---:|:---:|
| Запустить воркфлоу | — | ✓ | ✓ | ✓ | ✓ | — |
| Подтвердить шаг воркфлоу | — | ✓ | ✓ | ✓ | ✓ | — |
| Управлять каталогом воркфлоу | ✓ | ✓ | — | — | — | — |
| Просматривать audit log | ✓ | ✓ | — | — | — | ✓ |
| Управлять workspace и пользователями | ✓ | ✓ | — | — | — | — |
| Создавать workspace | ✓ | — | — | — | — | — |
| Экспортировать отчёты | ✓ | ✓ | — | — | — | ✓ |

### Архитектурные соображения

- Workspace-изоляция: tenant_id на каждой записи + middleware проверки принадлежности
- RBAC проверяется на каждый API-запрос; роли назначаются в рамках workspace
- Platform Admin действует вне workspace-контекста с системными правами
- Коммерческой тарификации нет — внутренний инструмент организации

## Functional Requirements

### Управление воркфлоу

- **FR1:** Аналитик, Team Lead, Architect или Developer может запустить воркфлоу из каталога с указанием входных параметров
- **FR2:** Team Lead может остановить выполняющийся воркфлоу
- **FR3:** Пользователь может просматривать список запущенных воркфлоу и их текущий статус
- **FR4:** Пользователь может повторно запустить воркфлоу с теми же зафиксированными версиями компонентов

### Движок выполнения воркфлоу

- **FR5:** Система выполняет воркфлоу строго в соответствии с зафиксированными версиями скиллов, промптов и rules на момент запуска
- **FR6:** Авторизованный пользователь может просматривать промежуточный артефакт на каждом шаге до подтверждения
- **FR7:** Team Lead, Architect, Analyst или Developer может подтвердить или отклонить результат шага воркфлоу
- **FR8:** Система сохраняет снимок версий всех компонентов воркфлоу (скиллы, rules, промпты, модель LLM) на момент запуска
- **FR9:** Система уведомляет участников о смене статуса шага или завершении воркфлоу
- **FR37:** Пользователь может перезапустить отдельный шаг при ошибке выполнения без перезапуска всего воркфлоу

### Управление артефактами

- **FR10:** Пользователь может просматривать все артефакты, сгенерированные в рамках воркфлоу
- **FR11:** Система связывает каждый артефакт с версией воркфлоу, скиллов и rules, которые его породили
- **FR12:** Пользователь может экспортировать артефакт в стандартном формате (Markdown, JSON)
- **FR13:** Система автоматически сохраняет финальные артефакты в подключённый Git-репозиторий
- **FR14:** *(Growth)* Пользователь может сравнить артефакты двух запусков одного воркфлоу

### Каталог воркфлоу

- **FR15:** Platform Admin может создавать, редактировать, публиковать и удалять воркфлоу в каталоге организации
- **FR16:** Team Lead может управлять воркфлоу в каталоге своего workspace
- **FR17:** Пользователь может просматривать доступные воркфлоу с описанием, параметрами и историей версий
- **FR18:** Система сохраняет полную историю версий воркфлоу; опубликованные версии неизменяемы
- **FR39:** Пользователь может искать воркфлоу в каталоге по названию и тегам

### Управление Workspace и пользователями

- **FR19:** Platform Admin может создавать и удалять workspace'ы
- **FR20:** Platform Admin и Team Lead могут приглашать пользователей в workspace и назначать роли
- **FR21:** Platform Admin и Team Lead могут изменять роли пользователей в workspace
- **FR22:** Система разграничивает данные между workspace'ами — пользователи видят только данные своего workspace
- **FR23:** Пользователь может аутентифицироваться с помощью логина и пароля *(Growth: LDAP/AD, SAML SSO)*

### Dashboard и аналитика

- **FR24:** Team Lead и Platform Admin могут просматривать сводную статистику запусков по workspace
- **FR25:** Пользователь может просматривать детальный статус запуска с разбивкой по шагам
- **FR26:** Team Lead может идентифицировать шаги воркфлоу с наибольшим количеством ручных исправлений
- **FR27:** Platform Admin и Team Lead могут экспортировать данные метрик в стандартном формате

### Уведомления

- **FR38:** Пользователь получает in-app уведомление, когда требуется его действие (подтверждение шага воркфлоу)

### Интеграции

- **FR28:** Система поддерживает интеграцию с self-hosted GitLab CE/EE для хранения и версионирования артефактов
- **FR29:** Система поддерживает интеграцию с self-hosted Bitbucket Server для хранения и версионирования артефактов
- **FR30:** *(Growth)* Система поддерживает интеграцию с публичными Git-провайдерами (GitHub, GitLab.com)
- **FR31:** *(Growth)* Система предоставляет API для запуска воркфлоу из внешних систем

### Аудит и трассируемость

- **FR32:** Система автоматически записывает все действия пользователей в воркфлоу в неизменяемый audit log
- **FR33:** Platform Admin, Team Lead и Auditor могут просматривать audit log с фильтрацией по дате, пользователю, workspace и воркфлоу
- **FR34:** Platform Admin, Team Lead и Auditor могут экспортировать записи audit log
- **FR35:** Auditor имеет read-only доступ к данным платформы без возможности запуска или изменения
- **FR36:** По любому артефакту система отображает полную цепочку трассируемости: воркфлоу → шаг → версии компонентов → пользователь → время

## Non-Functional Requirements

### Производительность

- **NFR1:** Интерактивные действия пользователя в UI (навигация, фильтрация, подтверждение шага) завершаются за < 1 с при нормальной нагрузке
- **NFR2:** Запуск воркфлоу и выполнение шагов — асинхронные операции; система уведомляет по завершению без индикации в реальном времени
- **NFR3:** API-запросы без LLM-вызова возвращают ответ за < 500 мс

### Безопасность

- **NFR4:** Все данные передаются по TLS 1.2+
- **NFR5:** Данные каждого workspace изолированы на уровне БД — tenant_id обязателен для всех записей
- **NFR6:** Аккаунт блокируется после 5 последовательных неудачных попыток входа
- **NFR7:** Секреты и credentials хранятся в зашифрованном виде и не логируются
- **NFR8:** RBAC-проверка выполняется на каждый API-запрос без исключений

### Масштабируемость

- **NFR9:** Система поддерживает до 50 активных workspace'ов без деградации производительности
- **NFR10:** Система поддерживает до 1000 параллельных запусков воркфлоу одновременно
- **NFR11:** Горизонтальное масштабирование компонентов возможно без изменения архитектуры

### Надёжность

- **NFR12:** Доступность платформы — не менее 99% в месяц (≤ 7.2 ч простоя)
- **NFR13:** Данные audit log неизменяемы и не удаляются при ошибках системы
- **NFR14:** При сбое шага воркфлоу состояние запуска сохраняется для перезапуска с прерванного шага (FR37)
