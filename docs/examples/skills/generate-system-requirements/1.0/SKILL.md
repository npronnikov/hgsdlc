# Generate system requirements

## Goal
Превратить запрос пользователя (и ответы на уточняющие вопросы, если они есть)
в структурированные, верифицируемые требования с привязкой к конкретным местам в коде.

---

## Шаг 1 — Анализ кодовой базы

Изучи затрагиваемые части системы:

1. **Текущее поведение**: как система работает сейчас в части, которую затронет запрос
2. **Модель данных**: схема таблиц, JPA-сущности, их поля и связи (OneToMany, ManyToOne и т.д.)
3. **API**: текущие эндпоинты, их контракты (запрос/ответ), статусы
4. **Бизнес-логика**: правила в сервисном слое, валидации, ограничения
5. **Авторизация**: текущие проверки доступа (@PreAuthorize, ручные проверки)
6. **Тесты**: что уже покрыто, какой стиль тестов принят в проекте
7. **Конфигурация**: настройки, переменные окружения, флаги фич

Зафиксируй список файлов, которые потребуют изменений.

---

## Шаг 2 — Синтез требований

На основе запроса, ответов на вопросы (если есть) и анализа кода сформулируй требования.
Каждое требование должно быть:
- **Конкретным**: понятно что делать
- **Верифицируемым**: можно написать тест или проверить вручную
- **Привязанным**: указан конкретный класс/метод/таблица/эндпоинт
- **Атомарным**: не объединять несколько несвязанных изменений в одно требование

---

## Формат вывода

Записать в файл `requirements.md`. Строго соблюдать структуру:

```markdown
# Требования к реализации

## Контекст запроса
{1 абзац: что именно запрошено и какие части системы затронуты}

## Список затрагиваемых файлов
- `src/.../SomeEntity.java` — добавить поле / изменить связь
- `src/.../SomeService.java` — новый метод / изменить логику
- `src/.../SomeController.java` — новый эндпоинт / изменить контракт
- `src/.../resources/db/changelog/...` — миграция схемы
- `src/.../SomeServiceTest.java` — новые тесты
...

---

**Где**: `ClassName#methodName` или `table_name.column`
**Правило**: {бизнес-правило, валидация, ограничение}
**Пример**: {конкретный пример корректного/некорректного ввода или результата}

### FR-2: ...

---

## Требования к API

### API-1: {эндпоинт}
**Метод и URL**: `POST /api/resource`
**Запрос**: {поля, типы, обязательность}
**Ответ (success)**: `201 Created` с {описание тела}
**Ответ (error)**: `400 Bad Request` если {условие}
**Авторизация**: {роль / условие доступа}

### API-2: ...

## Требования к модели данных

### DM-1: {изменение}
**Таблица**: `table_name`

**Кто может**: роль/условие
**Кто не может**: роль/условие

## Требования к тестированию

### TEST-1: {что покрыть}

- {happy path}
- {edge case 1}
- {edge case 2}

---

## Ограничения и риски

- {ограничение 1}: {почему и как обойти}
- {риск 1}: {описание и рекомендация}

## Критерии приёмки

- [ ] {проверяемый критерий 1}
- [ ] {проверяемый критерий 2}

- [ ] Нет регрессий в существующих тестах
```

---

## Ограничения

- Не писать код — только описание требований
- Ссылаться на реально существующие файлы и классы в проекте
- Если ответы на вопросы противоречат друг другу — указать противоречие явно
- Не добавлять требования, которые не следуют из запроса
