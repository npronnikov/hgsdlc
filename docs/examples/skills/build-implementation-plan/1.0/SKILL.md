# Build implementation plan

## Goal
Создать пошаговый план реализации, который позволит выполнить задачу предсказуемо,
с минимальным риском регрессий и ясными критериями готовности каждого шага.

---

## Шаг 1 — Прочитать требования

Прочитай `requirements.md` целиком. Выдели:
- Полный список затрагиваемых файлов
- Зависимости между требованиями (что должно быть готово раньше чего)
- Риски: что может сломать существующий функционал
- Что требует миграции данных

---

## Шаг 2 — Изучить целевые файлы

Открой и прочитай каждый файл из списка затрагиваемых. Запомни:
- Текущие сигнатуры методов и их контракты
- Зависимости классов (что инжектируется через конструктор)
- Существующие тесты — какие из них затронет изменение

---

## Шаг 3 — Построить порядок изменений

Определи правильный порядок шагов по принципу **«от фундамента к поверхности»**:

1. Изменения схемы БД (Liquibase)
2. Доменные сущности и value objects
3. Репозитории (новые методы / запросы)
4. Сервисный слой (бизнес-логика)
5. API-слой (контроллеры, DTO)
6. Тесты для каждого слоя
7. Конфигурация / ENV-переменные

Каждый шаг должен быть атомарным: после него проект компилируется и тесты проходят.

## Ограничения

- Не писать полный код — только скелеты, сигнатуры и ключевые детали
- Каждый шаг должен быть выполним за одну транзакцию работы (атомарность)
- Не планировать шаги, которые не следуют из требований
- Если требование неясно или противоречиво — указать это явно как риск
- Порядок шагов должен минимизировать количество мест, где проект «не компилируется»
