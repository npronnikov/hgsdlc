# Analyze project and generate clarifying questions

## Goal
Снять неопределённость запроса до начала реализации: изучить проект,
сформулировать вопросы, ответы на которые напрямую влияют на выбор решения.

---

## Шаг 1 — Изучение проекта

Перед формулировкой вопросов изучи кодовую базу:

1. **Точка входа**: `README.md`, `build.gradle` / `pom.xml`, `application.yml` / `application.properties`
2. **Структура пакетов**: найди основные слои (api, application, domain, infrastructure)
3. **Доменные сущности**: ключевые классы, их поля, связи между ними
4. **Существующие паттерны**: как обрабатываются ошибки, авторизация, валидация, транзакции
5. **API поверхность**: список контроллеров и их эндпоинтов — что уже есть
6. **Тест-покрытие**: есть ли тесты, какой их стиль (unit, integration, @WebMvcTest)
7. **Конфигурация**: базы данных, миграции Liquibase, внешние интеграции

Зафиксируй для себя: **что уже есть** и **что нужно изменить/добавить** согласно запросу.

---

## Шаг 2 — Анализ запроса

Исходя из запроса и изученной кодовой базы, определи:

- Какие компоненты системы затронет изменение?
- Что неясно: бизнес-правила, граничные случаи, scope изменений?
- Какие решения нужно принять до начала реализации?
- Где самый высокий риск ошибиться из-за недостатка информации?

---

## Шаг 3 — Генерация вопросов

Сформулируй от **10 до 15 вопросов**, покрывающих следующие категории:

| Категория | Что выяснять |
|-----------|-------------|
| **Scope** | Что именно должно измениться, что трогать нельзя |
| **Бизнес-правила** | Валидация, ограничения, граничные случаи |
| **API** | Формат запросов/ответов, HTTP-статусы, версионирование |
| **Данные** | Новые поля, миграции, обратная совместимость |
| **Интеграции** | Зависимости от других сервисов, события, очереди |
| **Авторизация** | Кто может вызвать новый функционал, какие роли |
| **Тестирование** | Ожидаемый уровень покрытия, типы тестов |
| **Развёртывание** | Нужна ли миграция данных, rollback-стратегия |

Каждый вопрос должен быть **конкретно привязан** к проекту и запросу — никаких абстрактных вопросов.

---

## Формат вывода

Записать результат в файл `questions.md`. Строго соблюдать формат:

```markdown
# Уточняющие вопросы

## Краткое резюме по проекту
{2–3 предложения: что за проект, какой стек, что уже реализовано из релевантного}

## Анализ запроса
{2–3 предложения: что запрашивается, какие части проекта затронуты, в чём основная неясность}

---

**Вопрос 1: {текст вопроса}**
- [ ] Вариант А: {конкретный вариант}
- [ ] Вариант Б: {конкретный вариант}
- [ ] Вариант В: {конкретный вариант}
- [ ] Другое: ___________

**Вопрос 2: {текст вопроса}**
- [ ] Вариант А: {конкретный вариант}
- [ ] Вариант Б: {конкретный вариант}
- [ ] Вариант В: {конкретный вариант}
- [ ] Другое: ___________

... (всего 10–15 вопросов)
```

---

## Ограничения

- Вопросы строго по делу — никаких вопросов ради количества
- Варианты ответа должны быть реалистичными и взаимоисключающими
- Не предлагать решения в вопросах — только прояснять требования
- Не писать никакого кода
- Вывод — только содержимое `questions.md`, ничего кроме него
